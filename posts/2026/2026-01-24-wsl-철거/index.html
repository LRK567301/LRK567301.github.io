caddy

wsl docker에서 쓰던 caddyfile 그대로 윈도우 네이티브 완벽. WSL의 네트워크가 문제인지 caddyfile을 주기적으로 읽을 때의 NTFS 성능이 문제인지 역방향 프록시 반응이 늦을 때가 한번씩 있었는데 윈도우로 넘기고 항상 빠르더라. 사용할 포트를 직접 방화벽에서 열어줘야 했던 WSL 미러 네트워크와 달리 윈도우 exe니까 알아서 방화벽 열리는 것도 편해.

FileBrowser Quantum

FileBrowser 시절은 사용할 경로를 담는게 좀 번거로웠던가? 여튼 설정이 살짝 귀찮았던 기억이 있는데 filebrowser.exe setup 하면 config 만들어주고 sources path 추가해주니 바로 되더라. cmd 팝업이 뜨니까 vb 스크립트로 숨겨서 작업 스케줄러 등록.

그냥 설치.

metube

북마클릿 포기하기 아까워서 고민했는데 지난번 pip 설치 시도가 시원찮았고 실제 사용 빈도를 생각하면 얼마나 쓴다고 서버를 구동해놓나 싶어졌다. yt-dlp 배치파일만 살짝 더 다듬었어. 정말 필요하면 파워쉘 스크립트로 txt 파일 하나 감시하게 하고 FileBrowser로 들어가서 url 집어넣어도 되니까.

wiresock + qbittorrent/firefox

순정 wireguard만으로 분할 터널링 될 줄 알았더니 Table = off 붙이는 순간 큐빗에서 네트워크 장치로 잡아줘도 통신이 안 되더라. 그래서 wiresock. 얘도 지난번에 AllowedApps는 되는데 AllowedIPs를 0.0.0.0/0으로 채울 수가 없어서 tunnlto였는데 이번엔 되네? 물론 한번엔 안 돼서 Enable Virtual Adaptor Mode하고 큐빗의 네트워크 어댑터도 설정했다. 그리고 gluetun http 프록시 전용 브라우저였던 파이어폭스도 AllowedApps로 묶으니까 wireproxy 필요도 없어.

근데 내부망 다른PC에서 크롬 Remote Torrent Adder 확장으로 IP로 접근이 안 된다? 데스크탑에서는 192.168.x.x:포트로 잘 붙고 방화벽도 다 열려있는데 그래. 방화벽 정책 추가, 큐빗 웹UI 보안 설정 변경 등 별짓 다 해보다 결국 얘도 FileBrowser/navidrome처럼 caddy 역방향 프록시 등록. Remote Torrent Adder 확장이 무조건 포트번호를 요구하길래 IP만 생각했는데 http/https 포트를 잊고 있었다.

그리고 큐빗 웹UI 설정에서 CSRF 보호/호스트 헤더 검사 체크 해제도 잊지 말자. 캐디가 중간에 낀 탓인지 페이지는 열리는데 마그넷 연결이 안 되더라.

Everything 1.5

마지막으로 무작위 사진 정렬을 위해 갈아탄 에브리씽 1.5 알파에 웹서버 플러그인까지 설치해서 마무리. 1.4랑 한동안 병행했는데 알파라도 언어만 영어일뿐 문제 없는 것 같아.

하룻밤 붙잡고 끙끙댔더니 윈도우 네이티브로 어지간히 다 해결됐다. winget 밖의 도구들이고 docker 유지보수관리의 편리함은 버려야 하지만 네이티브 개빠르고 가벼워. VmmemWSL 몇기가씩 먹는게 기본이었는데 큐빗 뜨는것조차 금방이고. WSL NTFS라는 모래주머니를 벗어던진 진짜 성능을 이제서야.

hugo처럼 작은 파일 많으면 모를까 대용량 다운로드는 괜찮겠지 했는데 생각보다 성능차이가 나는걸? WSL 안의 큐빗에서 파일 위치 이동을 시키면 수백MB짜리 파일도 이동 퍼센테이지가 올라가면서 완료됐는데 네이티브는 순식간. 진짜 모래주머니 제대로 달고 뛰고 있었구나.
