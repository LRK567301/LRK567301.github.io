WSL의 답답한 NTFS 접근 성능

젬민이랑 기술상담하다가 무설치 hugo는 괜찮은데 git 따로 설치하고 path 등록하고 이런 구조가 갑자기 거슬려서 WSL 통합 재도전. 전에는 내가 뭘 찾아야할지 키워드를 찾는 것부터 시작해야 했는데 젬민이는 그냥 내 생각을 뿌리면 대강의 솔루션을 한번에 제시하고 이어서 수정까지 가능하니까 고민하는 시간이 확 줄었다. AI 뒈졌으면을 심심하면 외치지만 기술적 문제 해결에는 너무 좋아서 쓰으으으읍 패배감……..

하지만 알려주는 대로 git 연동도 하고 hugo 빌드를 돌려봤더니 Pages 1176, Static files 2872 기준 Total in 223612 ms. WSL 멈춘 줄 알았다. 윈도우 네이티브 hugo는 30초면 될게 3분 이상. 그제서야 왜 내 hugo 폴더에 WSL로 돌려본 흔적이 남아있는지 기억났다. WSL의 NTFS 액세스 성능이 구려서였어. 큰 파일 한두개씩은 괜찮은데 자잘한 파일은 SMB랑 비슷한 느낌이라고 봐야겠다.

그럼 caddy가 한번씩 멍때리는 것도 비슷한 문제냐 했더니 역방향 프록시 정보가 담긴 Caddyfile에 항상 액세스하니까 부하가 걸릴 수 밖에 없다고. 이걸 매번 읽는다고? 진짜로? 싶긴 하지만 이참에 caddy도 다시 윈도우 네이티브로 돌렸다. 며칠 굴려보니 붙을 때마다 잘 살아있는데 기분 탓인지는 확실치 않아.
